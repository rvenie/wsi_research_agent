# ML System Design Doc - RND-researcher MVP 1

*Шаблон ML System Design Doc от телеграм-канала [Reliable ML](https://t.me/reliable_ml)*

## 1. Цели и предпосылки
### 1.1. Зачем идем в разработку продукта?
- Бизнес-цель: Автоматизировать поиск и сбор открытых датасетов с гистологическими слайдами (.svs, .tif) из публичных источников с анализом метаданных, научной значимости, доступности исходных данных и публикаций.
- Почему станет лучше: Экономия времени команд исследователей и инженеров, ускорение доступа к актуальным и качественным данным для ML-проектов в области цифровой патологии.
- Критерии успеха: Система может находить новые релевантные датасеты, извлекать по ним ключевые характеристики, корректно ссылаться на источники и отслеживать появление новых данных в известных статьях.

### 1.2. Бизнес-требования и ограничения
- Требования: Автоматизированный поиск, более 5 релевантных датасетов, отображение характеристик (объем, виды срезов, наличие исходных данных), актуальность, поддержка обновлений.
- Ограничения: Публичные источники, соблюдение лицензий на данные, ограниченный бюджет на вычисления, частичная интеграция научных публикаций.
- Бизнес-процесс пилота: ML- и DS-команды используют найденные датасеты в своих исследовательских и прикладных задачах; пилот считается успешным при обнаружении хотя бы 2-3 новых важных датасетов за итерацию.

### 1.3. Что входит в скоуп проекта/итерации, что не входит
- Входит: Автоматизация поиска датасетов и публикаций с открытым доступом, извлечение метаданных, создание простой базы/отчета с фильтрацией по атрибутам и ссылкам.
- Не входит: Сложная интеграция с коммерческими/приватными источниками, углубленный анализ изображений, разработка единого API для внешних сервисов.

### 1.4. Предпосылки решения
- Основная предпосылка: увеличивается количество открытых гистологических датасетов и научных публикаций, однако ручной мониторинг трудоемок; автоматизация необходима.
- Используем: только публичные ресурсы (Kaggle, TCGA, Zenodo, публикации PubMed, ResearchGate), агрегируем доступные ссылки и метаданные.

## 2. Методология
### 2.1. Постановка задачи
- Создать инструмент поиска и анализа открытых WSI-датасетов (whole slide images), автоматизировать сбор метаданных, определять научную новизну и стандартную структуру.

### 2.2. Блок-схема решения
- Блок-схема будет представлена в виде диаграммы в последующих итерациях. В MVP:
  1. Модуль поиска открытых датасетов (по ключевым источникам)
  2. Парсер публикаций и извлечение информации об исходных данных
  3. Сбор и обновление метаданных
  4. Валидация и отчет с характеристиками
  5. Проверка наличия исходных данных для известных статей

### 2.3. Этапы решения задачи
- MVP:
  1. Поиск и предварительный ручной сбор известных датасетов, разметка структуры и атрибутов для парсинга
  2. Разработка автоматизированного парсера датасетов (по исходным данным, размеру, форматам файлов, лицензии)
  3. Интеграция поиска публикаций по ключевым словам (PubMed, ResearchGate); выделение публикаций, где появились новые данные
  4. Сбор и регулярное обновление информации (cron/dag)

- Кейсы для базового теста: корректно заполняется база хотя бы по пяти открытым датасетам, собираются все ключевые атрибуты, автоматизированная проверка вариантов написания ключевых слов, обнаружение новых выпущенных публикаций по заданным авторам/статьям.

- Дальнейшее развитие: расширить поддержку новых источников, API для ML-методов извлечения изображений и разметки.

## 3. Подготовка пилота
### 3.1. Способ оценки пилота
- Вручную проверить полноту и качество поиска по контрольному списку (5+ новых датасетов, 80% извлеченных атрибутов соответствуют реальности, ссылки активны).

### 3.2. Критерии успешного пилота
- Найдены и корректно описаны минимум 5 датасетов и сведения о них, внедрено автоматическое отслеживание появления исходных данных по авторам.

### 3.3. Подготовка пилота
- Считаем ограничение по ресурсам: процедура получения информации должна занимать менее 12 часов в неделю на стандартном сервере/ноутбуке, допускается ручная валидация в первой версии.

---

## 4. Внедрение и инфраструктура (при развитии)
### 4.1. Архитектура решения
- Модули: парсер датасетов, анализ публикаций, база/отчет, планировщик обновлений.

### 4.2. Описание инфраструктуры
- Минимум: запуск скриптов по cron или через Airflow/Prefect на linux-сервере; поддержка ручного запуска и простого вывода в CSV/Markdown/SQLite.

### 4.3. Требования к работе системы
- SLA: не критично на этапе MVP. Пропускная способность — обновление основных источников раз в 2-4 недели.

### 4.4. Риски
- Огр. доступность новых источников, изменения в структуре сайтов, появление капчи/антибот-защит, нехватка хорошо описанных датасетов, частичное отсутствие исходных файлов.

---

## 5. Дополнительная информация
- Открытая проблема: автоматизация генерации новых ключевых слов/поисковых мест
- План на следующие итерации: поддержка английской и русской версий, валидация наиболее ценных датасетов, интеграция небольшой системы alert'ов о появлении новой информации.
